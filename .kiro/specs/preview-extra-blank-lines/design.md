# プレビュー余分空行バグ修正 設計ドキュメント

## 概要

VS Code webview プレビューにおいて、フェンスドコードブロックの末尾に余分な空行が表示されるバグを修正する。markdown-it は `token.content` に末尾の `\n` を必ず付加し、ブラウザがこれを `<code>` 要素内の余分な空行としてレンダリングする。現在の対策は行番号有効時のみ部分的に適用されており、行番号無効時には一切対策がない。また、`clipCodeToLineNumbers()` による JavaScript ワークアラウンドは webview のフォントレンダリング差異により不安定である。

修正方針は、markdown-it の fence レンダラーレベルで末尾 `\n` を確実に除去し、HTML/CSS レベルで行番号列とコード列の高さを一致させることで、JavaScript ワークアラウンドを不要にする。

## 用語集

- **Bug_Condition (C)**: webview プレビューでフェンスドコードブロックをレンダリングする際に、markdown-it が付加する末尾 `\n` が余分な空行として表示される条件
- **Property (P)**: コードブロックが末尾に余分な空行なしでレンダリングされ、行番号列とコード列の高さが一致する
- **Preservation**: PDF エクスポート、シンタックスハイライト、行番号の正確性、コピー動作が変更されないこと
- **`token.content`**: markdown-it がフェンスドコードブロックのコード部分を格納するプロパティ。常に末尾に `\n` が付加される
- **`wrapWithLineNumbers()`**: `src/parser/lineNumbers.ts` の関数。コードブロック HTML を行番号付きの2カラムレイアウトでラップする
- **`clipCodeToLineNumbers()`**: `media/preview.js` の関数。行番号列の高さに合わせてコード列の高さをクリッピングする JavaScript ワークアラウンド
- **`countLines()`**: `src/parser/lineNumbers.ts` の関数。コード文字列の行数をカウントする（末尾 `\n` は余分な行としてカウントしない）

## バグ詳細

### バグ条件

markdown-it は `fence` トークンの `token.content` に常に末尾 `\n` を付加する。この `\n` はブラウザの `<code>` 要素内で余分な空行としてレンダリングされる。現在の対策は以下の2点で不完全である：

1. `wrapWithLineNumbers()` の `\n</code></pre>` パターン除去は行番号有効時のみ適用される
2. 行番号無効時は末尾 `\n` を除去する処理が一切ない

**形式仕様:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { code: string, renderTarget: string }
  OUTPUT: boolean

  // markdown-it は常に token.content に末尾 \n を付加するため、
  // webview プレビューで表示されるすべてのフェンスドコードブロックが対象
  RETURN input.renderTarget = "webview-preview"
         AND input.code は fence トークンのコンテンツである
END FUNCTION
```

### 具体例

- **複数行コード（行番号有効）**: `console.log("a")\nconsole.log("b")\n` → 期待: 2行表示、実際: 3行目に空行が表示される
- **複数行コード（行番号無効）**: 同上 → 期待: 2行表示、実際: 3行目に空行が表示される
- **単一行コード**: `hello\n` → 期待: 1行表示、実際: 2行目に空行が表示される
- **空コードブロック**: `\n` → 期待: 空表示、実際: 1行の空行が表示される
- **末尾改行なしコード**: `hello` → 期待: 1行表示、実際: 1行表示（このケースは正常）
- **異なる言語のコードブロック（typescript, python, json, bash 等）**: すべて同様に末尾空行が表示される
- **言語指定なしのコードブロック**: 同様に末尾空行が表示される

## 期待される動作

### 保存要件

**変更されない動作:**
- PDF エクスポートのコードブロックレンダリングは現在と同様に正しく動作し続ける（`pre code { white-space: pre-wrap }` による）
- 行番号は正しい行数で表示され続ける（余分な行番号が追加されない）
- コードブロックのテキストコピー時に行番号は含まれない
- シンタックスハイライトは正しく適用され続ける
- 単一行コードブロックは正しく1行として表示され続ける

**スコープ:**
以下の入力はこの修正の影響を受けない：
- PDF エクスポートパスのレンダリング
- インラインコード（`` `code` ``）
- 非コードブロック要素（段落、見出し、リスト等）
- Mermaid / PlantUML / SVG のフェンスドブロック（`scanFencedBlocks` で別処理される）

## 仮説的根本原因

バグの分析に基づき、最も可能性の高い原因は以下の通り：

1. **markdown-it の末尾 `\n` 付加**: markdown-it の fence レンダラーは `token.content` に常に末尾 `\n` を付加する。これは markdown-it の仕様であり、変更できない。`<code>` 要素内でブラウザがこの `\n` を余分な空行としてレンダリングする

2. **行番号無効時の対策欠如**: `parseMarkdown.ts` の fence ルールオーバーライドは `options?.lineNumbers` が true の場合のみ適用される。行番号無効時は markdown-it のデフォルト fence レンダラーがそのまま使用され、末尾 `\n` が除去されない

3. **`wrapWithLineNumbers()` の不完全な除去**: 現在の正規表現 `\n</code></pre>` は末尾の `\n` を除去するが、markdown-it が生成する HTML の構造によっては完全に除去できないケースがある

4. **`clipCodeToLineNumbers()` の不安定性**: JavaScript による高さクリッピングは webview のフォントサイズ、行高さ、DPI 設定によって結果が異なり、安定した解決策ではない

## 正確性プロパティ

Property 1: Bug Condition - 末尾余分空行の除去

_For any_ フェンスドコードブロック入力（任意の言語、任意の行数、末尾改行あり/なし）において、修正後の fence レンダラーは `<code>` 要素内の末尾に余分な `\n` を含まない HTML を生成する SHALL。行番号の有効/無効に関わらず、コードブロックは余分な空行なしでレンダリングされる。

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - 行番号の正確性と既存機能の維持

_For any_ フェンスドコードブロック入力において、修正後のコードは以下を維持する SHALL：行番号列の行数がコードの実際の行数と一致すること、シンタックスハイライトが正しく適用されること、単一行コードブロックが正しく表示されること。

**Validates: Requirements 3.1, 3.2, 3.4, 3.5**


## 修正実装

### 必要な変更

仮説的根本原因の分析が正しいと仮定して：

**ファイル**: `src/parser/parseMarkdown.ts`

**関数**: `createMarkdownParser()` — fence ルールオーバーライド

**具体的な変更**:

1. **行番号無効時にも末尾 `\n` を除去する fence ルールを追加**: 現在は `options?.lineNumbers` が true の場合のみ fence ルールをオーバーライドしている。行番号無効時にも、レンダリング後の HTML から末尾の `\n` を除去する fence ルールを適用する

2. **末尾 `\n` 除去ロジックの統一**: 行番号有効/無効に関わらず、fence レンダラーの出力から `<code>` 要素内の末尾 `\n` を確実に除去する共通処理を実装する

**ファイル**: `src/parser/lineNumbers.ts`

**関数**: `wrapWithLineNumbers()`

**具体的な変更**:

3. **末尾 `\n` 除去の強化**: 現在の `\n</code></pre>` パターンに加え、markdown-it が生成する可能性のある他のパターンも確実に除去する。ただし、`parseMarkdown.ts` 側で事前に除去する方針であれば、この変更は不要になる可能性がある

**ファイル**: `media/preview.js`

**関数**: `clipCodeToLineNumbers()`

**具体的な変更**:

4. **`clipCodeToLineNumbers()` の削除または無効化**: HTML/CSS レベルで末尾 `\n` が除去されれば、JavaScript による高さクリッピングは不要になる。`initPreview()` および `message` イベントハンドラからの呼び出しも削除する

**ファイル**: `media/preview.css`

**具体的な変更**:

5. **CSS による高さ一致の保証**: `.ms-code-content` と `.ms-line-numbers` の高さが CSS grid レイアウトで自然に一致するよう、必要に応じてスタイルを調整する

**ファイル**: `examples/demo.md`

**具体的な変更**:

6. **コードブロックのバリエーション追加**: エッジケースカバレッジのため、以下の種類のコードブロックを追加する：
   - 単一行コードブロック
   - 空のコードブロック
   - 言語指定なしのコードブロック
   - 末尾改行が複数あるコードブロック
   - 非常に長い行を含むコードブロック
   - 異なる言語（Go, Rust, SQL, Dockerfile 等）のコードブロック
   - コメントのみのコードブロック
   - インデントが深いコードブロック

## テスト戦略

### 検証アプローチ

テスト戦略は2段階のアプローチに従う：まず、未修正コードでバグを実証するカウンターサンプルを表面化させ、次に修正が正しく機能し既存の動作を保持することを検証する。

### 探索的バグ条件チェック

**目標**: 修正実装前にバグを実証するカウンターサンプルを表面化させる。根本原因分析を確認または反証する。反証した場合は再仮説が必要。

**テスト計画**: markdown-it の fence レンダラーが生成する HTML を検査し、`<code>` 要素内の末尾に余分な `\n` が含まれることを確認する。未修正コードでこれらのテストを実行して失敗を観察し、根本原因を理解する。

**テストケース**:
1. **複数行コード（行番号有効）テスト**: 複数行コードをレンダリングし、コード列に余分な空行があることを確認（未修正コードで失敗）
2. **複数行コード（行番号無効）テスト**: 行番号無効で複数行コードをレンダリングし、余分な空行があることを確認（未修正コードで失敗）
3. **単一行コードテスト**: 単一行コードをレンダリングし、余分な空行があることを確認（未修正コードで失敗）
4. **空コードブロックテスト**: 空のコードブロックをレンダリングし、余分な空行があることを確認（未修正コードで失敗する可能性）
5. **異なる言語テスト**: TypeScript, Python, JSON, Bash, Go, Rust, SQL, Dockerfile 等の異なる言語でコードブロックをレンダリングし、すべてで余分な空行があることを確認
6. **言語指定なしテスト**: 言語指定なしのコードブロックで余分な空行があることを確認

**期待されるカウンターサンプル**:
- `<code>` 要素内の末尾に `\n` が含まれ、ブラウザが余分な空行としてレンダリングする
- 行番号有効時: 行番号列とコード列の行数が不一致
- 原因: markdown-it の fence レンダラーが `token.content` に末尾 `\n` を付加

### 修正チェック

**目標**: バグ条件が成立するすべての入力に対して、修正後の関数が期待される動作を生成することを検証する。

**擬似コード:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := renderFence_fixed(input)
  ASSERT NOT hasTrailingNewlineInCode(result)
  IF input.lineNumbersEnabled THEN
    ASSERT lineNumberCount(result) = codeLineCount(result)
  END IF
END FOR
```

### 保存チェック

**目標**: バグ条件が成立しないすべての入力に対して、修正後の関数が元の関数と同じ結果を生成することを検証する。

**擬似コード:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT renderFence_original(input) = renderFence_fixed(input)
END FOR
```

**テストアプローチ**: 保存チェックにはプロパティベーステストが推奨される。理由：
- 入力ドメイン全体にわたって多数のテストケースを自動生成する
- 手動ユニットテストでは見逃す可能性のあるエッジケースを検出する
- 非バグ入力に対して動作が変更されないことの強い保証を提供する

**テスト計画**: まず未修正コードでの動作を観察し、次にその動作を捕捉するプロパティベーステストを作成する。

**テストケース**:
1. **行番号正確性保存**: 未修正コードで行番号が正しく生成されることを観察し、修正後も同様であることを検証
2. **シンタックスハイライト保存**: 未修正コードでハイライトが正しく適用されることを観察し、修正後も同様であることを検証
3. **単一行コードブロック保存**: 未修正コードで単一行が正しく表示されることを観察し、修正後も行が消えないことを検証
4. **コードブロックバリエーション保存**: 異なる言語、空ブロック、長い行等のバリエーションで動作が保存されることを検証

### ユニットテスト

- `parseMarkdown.ts` の fence ルールが末尾 `\n` を除去することを検証（行番号有効/無効の両方）
- `wrapWithLineNumbers()` が末尾 `\n` を含まない HTML を受け取った場合に正しく動作することを検証
- `countLines()` が各種コード文字列に対して正しい行数を返すことを検証
- 空コードブロック、単一行コードブロック、末尾改行なしコードブロックのエッジケース
- 異なる言語（TypeScript, Python, JSON, Bash, Go, Rust, SQL, Dockerfile, 言語指定なし）のコードブロック

### プロパティベーステスト

- ランダムなコード文字列を生成し、修正後の fence レンダラーが末尾余分空行を含まないことを検証
- ランダムなコード文字列と行数で `wrapWithLineNumbers()` の行番号正確性を検証
- ランダムな言語指定でシンタックスハイライトの保存を検証
- 単一行から多数行までのランダムなコードブロックで、行番号列とコード列の行数一致を検証

### 統合テスト

- `createMarkdownParser()` で完全な Markdown ドキュメントをレンダリングし、すべてのコードブロックに余分な空行がないことを検証
- 行番号有効/無効の切り替えで正しくレンダリングされることを検証
- `examples/demo.md` の各種コードブロックバリエーションが正しくレンダリングされることを検証
- PDF エクスポートパスのレンダリングが変更されないことを検証
